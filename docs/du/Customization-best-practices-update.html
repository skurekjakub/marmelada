      Customization best practices update

      Based on notes from consultants and support.

      Database modifications

      • modification of system tables (add, remove fields)

        • modify only classes which are 'customizable'

        • when you need to modify not customizable classes - how to do that?

          • SAFE WAY: create a custom table (custom module class) and reference extra fields/data via a reference key (e.g. RoleID, RoleGUID)

            • What happens during synchronization between environments with staging or import/export)?

              • If this extended table is a custom module class table and TYPEINFO is properly defined, staging & import/export should work just fine with the reference keys/fields

            • NOTE: You might need to implement custom API (e.g. object query) in custom <>InfoProvider to retrieve/work with data from both the original class and custom class and always call this method. If not possible, you can as well override the original class <>InfoProvider and add extra functionality.

              Candidate to add to best practices! For adding fields to classes that are not customizable.

          • FAST WAY (not officially supported)

            • modify the class directly by enabling the customizable option and make note(s) about your changes.

      • changing size and type of system fields (e.g. increasing size of NodeAliasPath, RoleDisplayName, varchar to char leading into TRUNCATE issues)

        • Look for alternative solution (e.g. using DocumentURLPath instead of NodeAliasPath if looking for longer field) since hardcoded things are in many cases impossible to modify without source code version.

          Candidate - mention searching for alternative solutions / contact support.

      • changing database schema (e.g. dbo)

        • This should not be an issue at all as long as they ensure proper permissions / unified schema for the SQL account used in Kentico's connection string, so we allow customers to change DB schema of DB objects (tables, views etc.). It seems like we are using setting "CMSDBObjectOwner" for online forms and generally when creating a new class (page type, custom module class and custom table).

          Add info about dbo schema - based on connections string / setting. - find out info and add

      • customization of database properties (collation etc.)

        • Do not modify collation

        • Everything else is up to customer, since there are so many options how to configure DB. For example, if they set isolation level to something that performs way better from performance perspective (ReadUncommitted), they they will need to live with the fact that the data might not be 100% accurate which could cause some possible issues in Kentico. Best practice would be to validate they DB properties changes with our tech.support. Ideally when in communication with technical support, provide them also with customizations to DB properties.

          yes - isolation level etc. Add mention (somewhere) to ask Kentico support if their configuration is compatible.

      • adding foreign keys for custom tables (possible SYNC issue(s))

        • Use module classes instead when you need foreign keys, unless you need to get fast access to UI which custom tables module has.

          add note to best practices that foreign keys are supported only for custom module classes

      • using not supported field data type (for example XML)

        • There should be an option to add custom data type in general by calling DataTypeManager.RegisterDataTypes, but this is not described in the documentation. This would the the best practice compared to changing data type directly on DB level. This is of course not-transferable between different Kentico instances, so all instances would need to contain this customization.

          Research and add a page to the documentation describing the scenario. (basic info + example from consultants)

          Separate page, link from the general customization best practices.

      • adding new fields with required flag ('not null') running into SYNCH issues + having different field data type (int and string) in different environments (STG, PROD) for the same object (possible SYNC issue(s))

        • When table already contains some data and you are adding a new field, best practice is to always make this field not required -> allow null.

          Import/Export, Staging doc note, not in customization best practices.

      • have unique prefix in general (custom fields, custom tables / objects)

      • adding/removing/updating data via SQL

        • modification of system table's data (e.g.UI elements) - avoid in all cases

        • Best practice would be to use Kentico API (might be slow for large amount of data). Speed up API processing with CMSActionContext object.

          Add a new page / subject to the Performance optimization section – Importing / Managing large volumes of Kentico data

          - Use the Kentico API if possible, use CMSActionsContext to disable additional tasks (staging, event logging, continuous synchronization)

          - Bulk insert API? https://kentico.atlassian.net/wiki/display/CON/Bulk+Insert+feature

      • list all of the built-in system page types (for example CMS.Blog*)

        • do not modify system page types

          Extend/check the list of "system objects" in the best practices. For example the CMS.Blog* page type.

          + check other "hardcoded" system objects in the solution

      • IDEAS

        • define diagrams explaining database structure for a specific module

          • rather than understanding database structure, they should be interested in API, specifically references between classes, assemblies / DLLs etc. which you can get in VS by generating UML class diagram.

            Consultants will try to prepare a solution to generate diagrams for specific modules (e.g. blogs). TWs then need to integrate the results into the docs.

            Related to the Object tree reference project, but that is still pretty far off.

        • missing list of all web.config keys

          • what is not published/listed in documentation should not be provided to customers since it might be a prototype or something that we might remove in a future / is not tested.

          • NOTE: You can store settings in web.config, could.config or on database level in CMS_SettingsKey table. The priority is not deterministic, depends from case to case.

            Add note to web.config reference or general customization practices. Remove old web.config keys that are not documented - try to look for an equivalent setting in the Settings application

            *source control requirement for setting history? solution - use continuous integration for settings

      File system modifications

      • modification of system files (e.g.g CKEditor, PortalTemplate.aspx) in case there is no customization option / extension point for such module/feature/functionality. Create a copy or modify system file?

        • if copy, then hotfix/upgrade will overwrite the original file, but not your custom file, so you don't know what was changed unless yo have a note about your customization?

        • if original, then you know what was hotfixed/upgraded

          • maybe keep a copy of the original file with extension .old82_15 (version 8.2, hotfix 15)

          • what if source control is used?

            • maybe just keep comments in the code and when Kentico creates the .new file, then you can compare

          • if you completely change the original file, it might be better to have the copy of original.

            Note - For small customizations, it may be better to edit Kentico project files directly (try to update the customization best practices in a clear way)

      • API changes, some methods are removed and there is no replacement or not published online

        • In past there was some API that was public and should not be, which is our fault.

      • modification of codebehind and/or codefile of custom web part layout

        • when customizing web part markup (.ascx) do you recommend to create a virtual layout of modify the file on a file system?

        • Use custom layout only to reorganize layout, not to write a custom code (e.g. <script runat="server"> ... </script>)

          No use of inline code in web part layouts - add to web part documentation + best practices

          In these cases, create a custom web part (link)

      • best practices for using Bootstrap? - Ondra Vasil, then add customization BPs (similar to jQuery best practices)
        use of PortalContext.ViewMode in custom code to prevent conflicts of Kentico UI with custom CSS and/or JS

        Avoid problems with Kentico UI styles conflicting with site styles

        Try to avoid conflicts, or if necessary use PortalContext.ViewMode conditions (macros or custom code) to add styles/JS files

      • not calling base implementation of built-in methods when customizing some provider

        • developer needs to know what he is doing, if he does not know what the base class is doing, it is better to call the base implementation and then add custom code.

          Provider customization - call the base method unless you require complete customization or you know exactly what you are doing.

      • Custom class in App_Code (OLD_App_Code) vs. custom DLL?

        • Why is custom DLL preferred according to the documentation?

          • the main benefits are performance, test-ability, transfer-ability (e.g. it is included in a custom module, nuget package)

            Consider adding the reasons/benefits to the best practices. Performance/compiling etc.

      • where to store custom files in folders like CMSWebParts etc. or in completely custom location like MyProject/MyWebParts etc.

        • follow the documentation for custom modules and keep custom files under module recommended location. If you do not have a custom module, create one. Global location is not recommended, because in a future everything should be considered a module and for example some web part will always belong to only one module, not to multiple modules, so it makes sense to keep in in module recommended location.

          Add files to a custom sub-folder when possible, then use the structure underneath.

          If possible create a custom module and use the module structure

          + exceptions for special cases (web parts, form controls)

      • how to upgrade customers with source code? (not supported)

        Consider adding a quick process for source code upgrades - merge source code versions, upgrade the database. Add note to upgrade instructions.

      • use custom storage provider to change which file is loaded by system (file1.aspx vs file2.aspx)

        • this is not a best practice, because this could be an issue due to .NET handling the file (e.g. precompilation).

      • IDEAS

        • have a list of web parts that were removed and replacement / option

        • can we list all of the loaded custom providers, classes etc. on application start in the event log or simply show in the system application?

        • system application could tell you if you are using web app, web site, azure project or precompiled web site

        • define optimization web.config keys (e.g. disable Mobi 51Degrees)

        • importing data/files from old versions to new one, can overwrite changes

        • upgrade with opened VS, could this cause an issue(s)?

        • resign macros after import/export or synchronization with staging in certain cases

        • when cloning some objects (e.g. page template), be aware that import/export / staging can overwrite your objects because it uses the same GUID
